## 2. สรุปรายละเอียดของความต้องการ (Detail Requirements)

### 2.1 Functional Requirements

#### 2.1.1 Module 1 — การวิเคราะห์ต้นทุนโลจิสติกส์ของผู้ประกอบการนำเข้าในเบื้องต้น (Logistics Import Cost Estimation)

* **REQ1.1 รับข้อมูลนำเข้าจากผู้ใช้**: ระบบต้องรับข้อมูลสินค้าที่จะนำเข้า ประเทศต้นทางและท่าส่งออก ประเทศปลายทาง (ประเทศไทย) และท่านำเข้า Incoterm ถิ่นกำเนิดสินค้า รูปแบบการขนส่ง (อากาศ ทะเล บก) พิกัดศุลกากรระดับ 8 หลักพร้อมรหัสสถิติ 3 หลัก ราคาสินค้า ค่าขนส่ง และค่าประกันภัย (หน่วยบาท)
* **REQ1.2 คำนวณราคา CIF**: ระบบต้องคำนวณและแสดงราคา CIF ของสินค้านำเข้า (บาท)
* **REQ1.3 คำนวณภาษีนำเข้าทั้งหมด**: ระบบต้องคำนวณอัตราภาษีและจำนวนภาษีที่เกี่ยวข้อง ได้แก่ ภาษีศุลกากร ภาษีมูลค่าเพิ่ม ภาษีสรรพสามิต (ถ้ามี) และภาษีเพื่อมหาดไทย (ถ้ามี)
* **REQ1.4 ประมาณค่าใช้จ่ายโลจิสติกส์ปลายทาง**: ระบบต้องแสดงค่าใช้จ่ายด้านโลจิสติกส์ในประเทศปลายทาง เช่น ค่าเอกสาร ค่าใช้จ่ายภายในท่าเรือหรือสนามบิน และค่าบริการผู้ให้บริการขนส่งระหว่างประเทศ
* **REQ1.5 แนะนำข้อกำหนดการนำเข้า (OGA / Permit)**: ระบบต้องแนะนำว่าสินค้าต้องขอใบอนุญาตจากหน่วยงานควบคุมหรือต้องเสียภาษีอื่นที่เกี่ยวข้องหรือไม่
* **REQ1.6 เปรียบเทียบสิทธิประโยชน์ FTA**: ระบบต้องแนะนำสิทธิประโยชน์ลดภาษีนำเข้าภายใต้เขตเสรีการค้าที่เกี่ยวข้อง (MVP: ACFTA, AFTA/ATIGA, RCEP) พร้อมอัตราภาษีและประมาณการยอดภาษีที่ประหยัดได้ (บาท)
* **REQ1.7 ประมาณระยะเวลาการขนส่ง**: ระบบต้องแสดงระยะเวลาการขนส่งโดยประมาณ (วัน)
* **REQ1.8 แนะนำผู้ให้บริการ**: ระบบต้องแนะนำผู้ให้บริการขนส่งสินค้าระหว่างประเทศและตัวแทนออกของที่รับอนุญาต

#### 2.1.2 Module 2 — การวิเคราะห์สินค้านำเข้าเพื่อแนะนำพิกัดศุลกากร (Tariff Intelligence)

* **REQ2.1 รับข้อมูลสินค้าเพื่อตีพิกัด**: ระบบต้องรับรูปภาพของสินค้านำเข้า ข้อมูลทางเทคนิค พิกัดศุลกากรจากประเทศต้นทาง และพิกัดที่คาดว่าจะเป็น (ถ้ามี)
* **REQ2.2 แนะนำพิกัดศุลกากร**: ระบบต้องประมวลผลและแนะนำพิกัดศุลกากรของสินค้านำเข้าระดับ 6–8 หลัก พร้อมระดับความมั่นใจ (confidence)
* **REQ2.3 ฐานข้อมูลพิกัด (Tariff Intelligence DB)**: ระบบต้องมีฐานข้อมูลพิกัดศุลกากรสำหรับการค้นหาและจับคู่ โดยขอบเขต MVP ครอบคลุม HS Code ยุทธศาสตร์ 200 รายการ และอัตราภาษี MFN + ACFTA + AFTA + RCEP

#### 2.1.3 Module 3 — การเตรียมความพร้อมการจัดทำใบขนสินค้าขาเข้า (Customs Entry Preparation)

* **REQ3.1 รับเอกสารประกอบการนำเข้า**: ระบบต้องรับบัญชีราคาสินค้า (Commercial Invoice) ใบกำกับหีบห่อ (Packing List) ใบตราส่ง (B/L / AWB / Truck Waybill) พิกัดศุลกากรที่ใช้ เอกสารประกอบการสำแดงราคาศุลกากร ชื่อผู้นำของเข้าพร้อมเลขประจำตัวผู้เสียภาษี และเอกสารสิทธิประโยชน์ (เช่น ใบรับรองถิ่นกำเนิดสินค้า)
* **REQ3.2 สกัดและจัดเตรียมข้อมูลใบขน**: ระบบต้องประมวลผลและแสดงผลลัพธ์สำหรับจัดทำใบขนสินค้าขาเข้าอย่างน้อยดังนี้:
  * เลขที่ใบตราส่ง ชื่อยานพาหนะ เลขที่บัญชีราคาสินค้าและวันที่
  * เครื่องหมายและเลขหมายหีบห่อ จำนวนหีบห่อ
  * ประเทศกำเนิด ประเทศต้นทางที่บรรทุก ท่าเรือที่นำเข้า
  * พิกัดศุลกากรและรหัสสถิติ รหัสสิทธิพิเศษ
  * ราคาของ (หน่วยเงินต่างประเทศและบาท) น้ำหนักสุทธิ อัตราแลกเปลี่ยน
  * อากรขาเข้า ค่าธรรมเนียม ภาษีอื่นๆ อัตราและจำนวนภาษีสรรพสามิต ภาษีเพื่อมหาดไทย ภาษีมูลค่าเพิ่ม
  * ชนิดของ ตราสินค้า ปริมาณสินค้า
  * เลขที่ใบอนุญาตหรือเอกสารที่เกี่ยวข้อง และเลขที่เอกสารสิทธิประโยชน์ (เช่น Form E / D / RCEP)
* **REQ3.3 Pre-flight verification**: ระบบต้องตรวจสอบความครบถ้วนและความสอดคล้องของเอกสารก่อนยื่นใบขน และแสดงสรุปความพร้อมพร้อม foreshadow ต้นทุนภาษี
* **REQ3.4 ส่งต่อข้อมูลสู่ระบบยื่นใบขน**: ระบบต้องจัดเตรียมข้อมูลในรูปแบบที่พร้อมส่งต่อไปยังซอฟต์แวร์/NSP (data passthrough / JSON payload) เพื่อลดการกรอกข้อมูลซ้ำ

#### 2.1.4 Module 4 — การตรวจสอบความถูกต้องในการนำเข้าย้อนหลัง (Post-Audit / Import Entry Verification)

* **REQ4.1 รับเอกสารเพื่อตรวจสอบย้อนหลัง**: ระบบต้องรับใบขนสินค้าขาเข้า Commercial Invoice Packing List ใบตราส่ง พิกัดศุลกากรที่ใช้ เอกสารประกอบราคาศุลกากร และเอกสารสิทธิประโยชน์
* **REQ4.2 ตรวจหาความคลาดเคลื่อน**: ระบบต้องแสดงผลลัพธ์ที่คลาดเคลื่อนหรือผิดพลาดในการสำแดงในใบขนสินค้าขาเข้า (เช่น การตีพิกัด การสำแดงราคา การใช้สิทธิ FTA ความครบถ้วนของเอกสาร)
* **REQ4.3 แนะนำแนวทางแก้ไขเบื้องต้น**: ระบบต้องให้ข้อแนะนำสำหรับผู้ประกอบการในการดำเนินการด้านศุลกากรในเบื้องต้นเมื่อพบความผิดพลาด
* **REQ4.4 เก็บประวัติการตรวจสอบ**: ระบบต้องจัดเก็บผลการตรวจสอบย้อนหลังเพื่ออ้างอิงและติดตาม (รองรับการเก็บข้อมูลอย่างน้อยตามระยะเวลาที่กฎหมายกำหนด)

### 2.2 Non-Functional Requirements

* **REQ5.1 ความเสถียรของระบบ (System Stability)**: ระบบต้องพร้อมใช้งานอย่างต่อเนื่อง โดยมี downtime ไม่เกิน 0.5% ต่อเดือน
* **REQ5.2 ประสิทธิภาพ (System Performance)**: ระบบต้องรองรับผู้ใช้งานพร้อมกันได้อย่างน้อย 100 sessions โดยไม่ทำให้เวลาตอบสนองของหน้าจอหลักลดลงอย่างมีนัยสำคัญ
* **REQ5.3 ความปลอดภัยของข้อมูล (Data Security)**: ข้อมูลผู้ใช้และเอกสารนำเข้าต้องถูกเข้ารหัสในการจัดเก็บและรับส่ง มีการควบคุมสิทธิ์การเข้าถึง และใช้ SSL/TLS บน production
* **REQ5.4 ความถูกต้องของข้อมูลภาษี**: อัตราภาษีและสิทธิ FTA ใน MVP ต้องอ้างอิงจากแหล่งข้อมูลทุติยภูมิที่อนุมัติ และมีการระบุวันที่อัปเดตข้อมูลให้ผู้ใช้ทราบ
* **REQ5.5 การใช้งานบนอุปกรณ์ (Responsive Web)**: ระบบต้องใช้งานได้บนเว็บเบราว์เซอร์หลัก และรองรับการแสดงผลแบบ responsive บนหน้าจอมือถือ/แท็บเล็ต (ไม่รวม Mobile Native App ใน Phase 1)

## 5. การอัพเดทสถานะโครงการ (Project Status Updates)

ในการดำเนินโครงการนี้ จะมีการอัพเดทสถานะโครงการให้แก่เจ้าของโครงการทุก 1 เดือน โดยจะจัดทำรายงานความคืบหน้า (Progress Report) ที่ประกอบด้วย:

* ความคืบหน้าของงานที่เสร็จสิ้นตามแผนการดำเนินงาน (เทียบกับ Project Schedule)
* ปัญหาหรืออุปสรรคที่พบเจอในระหว่างการดำเนินโครงการ
* การปรับแผนหากมีการเปลี่ยนแปลงขอบเขตหรือเวลาในการดำเนินงาน (ผ่านกระบวนการ Change Request)
* ความเสี่ยงที่เกิดขึ้นใหม่ และการแก้ไขหรือจัดการความเสี่ยงนั้น ๆ

การอัพเดทสถานะนี้จะทำให้เจ้าของโครงการมีข้อมูลที่ครบถ้วนและสามารถติดตามความคืบหน้าได้ตลอดระยะเวลาของโครงการ เพื่อให้มั่นใจว่าโครงการจะดำเนินไปตามเป้าหมายและงบประมาณที่วางไว้

## 7. การสำรองข้อมูลและการกู้คืน (Backup & Recovery)

เพื่อให้การดำเนินงานของโครงการเป็นไปอย่างราบรื่นและมีความปลอดภัย ระบบจะต้องมีการสำรองข้อมูลและการกู้คืนอย่างเหมาะสมเพื่อป้องกันความสูญเสียของข้อมูลที่สำคัญ

### รายละเอียดการสำรองข้อมูล

* **ข้อมูลที่สำรอง**: ซอร์สโค้ด ฐานข้อมูล (รวม Tariff Intelligence DB) เอกสารการออกแบบระบบ เอกสารข้อกำหนด (SRS) และข้อมูลการทดสอบระบบ (Test Data)
* **วิธีการสำรอง**: สำรองผ่านเครื่องมือที่รองรับการทำงานอย่างต่อเนื่อง ไปยังพื้นที่จัดเก็บบนคลาวด์ที่มีการควบคุมการเข้าถึง
* **เครื่องมือที่ใช้**: ใช้ Git สำหรับการควบคุมเวอร์ชันของซอร์สโค้ด และใช้ระบบจัดเก็บเอกสารบนคลาวด์ (เช่น Google Drive) สำหรับเอกสารโครงการ
* **ผู้รับผิดชอบ**: ________________________ (DevOps / PR) เป็นผู้ตั้งค่าและตรวจสอบความสมบูรณ์ของข้อมูลสำรอง โดยมี Administrator ตรวจสอบเมื่อต้องกู้คืน
* **ความถี่ในการสำรองข้อมูล**:
  * **สำรองข้อมูลรายเดือน (Monthly Backup)**: ทำการสำรองข้อมูลทุกเดือน โดยจัดเก็บไว้บนคลาวด์ระยะยาวเพื่อความปลอดภัยและเข้าถึงได้ง่าย
  * ซอร์สโค้ด: push และเก็บ history บน Git remote เป็นประจำทุกครั้งที่มีการเปลี่ยนแปลงสำคัญ

### แผนการกู้คืนข้อมูล (Recovery Plan)

* **การกู้คืนข้อมูล**: ในกรณีฉุกเฉินหรือสูญเสียข้อมูล ทีมผู้ดูแลระบบจะกู้คืนจากชุดสำรองบนคลาวด์ และยืนยันความสมบูรณ์ก่อนคืนเข้าสู่ระบบหลัก โดยสำรองชั่วคราว (Temporary Backup) ก่อนดำเนินการทุกครั้ง
* **เครื่องมือสำหรับการกู้คืน**: Git สำหรับซอร์สโค้ดเวอร์ชันล่าสุด และระบบคลาวด์เอกสารสำหรับเอกสารโครงการ
* **ความถี่ในการทดสอบการกู้คืน**: ทดสอบแผนการกู้คืนอย่างน้อยรายเดือน เพื่อให้แน่ใจว่ากระบวนการทำงานได้จริง
