## 1. บทนำ (Introduction)

TradeBridge Digital เป็นระบบช่วยบริหารจัดการข้อมูลภาษีและต้นทุนการนำเข้า ด้วยเทคโนโลยีดิจิทัล เพื่อให้ผู้นำเข้า — ทั้งบุคคลธรรมดา บริษัท หรือ broker — สามารถสมัครใช้งาน เลือกแพ็กเกจ ชำระเงินเพื่อปลดล็อกฟีเจอร์ แล้ววิเคราะห์ต้นทุนก่อนสั่งซื้อ แนะนำพิกัดศุลกากร เตรียมข้อมูลใบขนสินค้าขาเข้า และตรวจสอบความถูกต้องของการนำเข้าย้อนหลังได้อย่างมีประสิทธิภาพภายใต้ขอบเขต MVP ที่ตกลงร่วมกัน

## 2. ความต้องการของระบบ (System Requirements)

### 2.1 ภาพรวมความต้องการของระบบ (System Requirement Overview)

ระบบออกแบบมาเพื่อให้ผู้นำเข้า / ผู้ประกอบการ / broker สามารถ:

* สมัครบัญชี เลือกแพ็กเกจ และชำระเงินผ่าน Payment Gateway เพื่อเข้าถึงฟีเจอร์ตามสิทธิ์
* วิเคราะห์ต้นทุนโลจิสติกส์และภาษีนำเข้าก่อนตัดสินใจสั่งซื้อ
* แนะนำพิกัดศุลกากรจากข้อมูลและรูปภาพสินค้า
* เตรียมและตรวจสอบเอกสารก่อนยื่นใบขน
* ตรวจสอบความถูกต้องของการนำเข้าย้อนหลัง (Post-Audit)

โดยส่งมอบในรูปแบบเว็บแอปพลิเคชันแบบ responsive

* ระบบเชื่อมต่อ **Payment Gateway** สำหรับการชำระค่าแพ็กเกจ พร้อมรับผลผ่าน callback/webhook

* ระบบอาจใช้บริการจัดเก็บไฟล์บนคลาวด์สำหรับการอัปโหลดเอกสารนำเข้า

## 3. ความต้องการทางซอฟต์แวร์ (Software Requirements)

### 3.1 ภาพรวมความต้องการทางซอฟต์แวร์ (Software Requirement Overview)

TradeBridge Digital พัฒนาในรูปแบบ **Web Application** แบบ responsive โดยมีองค์ประกอบหลักดังนี้:

* **Access Layer** สำหรับ Register/Login, Package Catalog, Payment และ Entitlement
* **Tariff Intelligence DB** สำหรับพิกัดและอัตราภาษีในขอบเขต MVP
* **Pre-Trade Intelligence / Duty Estimator** สำหรับการคำนวณต้นทุนและเปรียบเทียบ FTA
* **Core Business Logic Engine** สำหรับ Duty Calc, Green/Red rule-based และ FTA Savings
* **Customs Entry Preparation & Post-Audit workflows** สำหรับเตรียมใบขนและตรวจสอบย้อนหลัง

การคำนวณใน MVP ใช้แนวทาง **rule-based** (ไม่รวม ML Model / CBAM Engine ขั้นสูง)

### 3.2 ความสามารถของระบบ (Requirements Functionality)

ซอฟต์แวร์ต้องรองรับการทำงานดังนี้:

* รองรับผู้ใช้งานพร้อมกันได้อย่างน้อย **100 sessions**
* ข้อมูลผู้ใช้ เอกสารนำเข้า และข้อมูลการชำระเงินต้องถูกเข้ารหัส และมีการตรวจสอบสิทธิ์การเข้าถึง
* บังคับใช้ entitlement ตามแพ็กเกจก่อนเข้าถึงโมดูลธุรกิจ
* แสดงวันที่อัปเดตของข้อมูลอัตราภาษี/FTA ที่ใช้อ้างอิงในการคำนวณ
* จัดเก็บประวัติการวิเคราะห์ การเตรียมใบขน การตรวจสอบย้อนหลัง และการชำระเงินเพื่อการติดตาม

### 3.3 ส่วนต่อประสานภายนอก (External Interfaces)

* เชื่อมต่อ Payment Gateway สำหรับ checkout และยืนยันผลการชำระเงิน
* ใช้ช่องทาง JSON / data export สำหรับส่งต่อไปยังซอฟต์แวร์/NSP
* ใช้ Git และระบบจัดเก็บบนคลาวด์สำหรับการควบคุมเวอร์ชันซอร์สโค้ดและเอกสารโครงการ (ระดับกระบวนการพัฒนา)

## 4. ความต้องการที่ไม่ใช่การทำงานของระบบ (Non-functional Requirements)

### 4.1 ส่วนต่อประสานผู้ใช้ (User Interface)

ส่วนต่อประสานผู้ใช้ต้องครอบคลุม workflow ของ Access Layer (Register, Login, Package Offer, Checkout, Account/Subscription) และ 4 โมดูลหลัก รวมถึงแดชบอร์ดสรุป โดยออกแบบให้สื่อสารผลลัพธ์การคำนวณ checklist และคำแนะนำได้อย่างชัดเจน รายละเอียดหน้าจอจะอยู่ในเอกสาร Software Design (SDD)

### 4.2 การใช้งานง่าย (Usability and Ease of Use)

ระบบออกแบบให้ผู้ใช้ที่เป็นผู้นำเข้า/ตัวแทนออกของเข้าใจและใช้งานได้รวดเร็ว มีการแสดงผลลัพธ์พร้อมคำอธิบายสั้น ๆ และรองรับการใช้งานผ่านเบราว์เซอร์หลัก

### 4.3 ประสิทธิภาพ (Efficiency)

ระบบต้องตอบสนองได้เหมาะสมต่อการวิเคราะห์และประมวลผลเอกสารภายในขอบเขต MVP และรองรับการใช้งานพร้อมกันตามที่ระบุในข้อ 3.2

### 4.4 ความน่าเชื่อถือ (Reliability)

ระบบต้องมีความเสถียร โดยมี downtime ไม่เกิน **0.5% ต่อเดือน** และมีแนวทางการสำรอง/กู้คืนข้อมูลตามแผนโครงการ

### 4.5 ความปลอดภัย (Security)

* ใช้ SSL/TLS บนสภาพแวดล้อม production
* เข้ารหัสข้อมูลสำคัญทั้งระหว่างรับส่งและจัดเก็บตามแนวทางที่กำหนดในการออกแบบ
* ควบคุมสิทธิ์การเข้าถึงตามบทบาทผู้ใช้

## 5. ขอบเขตนอกระบบ (Out of Scope)

* Mobile Native Application
* ความตกลง FTA นอกเหนือจาก ACFTA, AFTA/ATIGA และ RCEP
* ML Model / predictive engine ขั้นสูง และ CBAM Engine
* การยื่นใบขนแทนกรมศุลกากรโดยตรง (ระบบเป็นชั้นเตรียมข้อมูลและตรวจสอบ)
* Multi-seat organization billing ขั้นสูง (เช่น หลายที่นั่งภายใต้ subscription เดียวแบบ enterprise hierarchy) — บัญชี MVP เป็นบัญชีผู้นำเข้าต่อผู้สมัครหนึ่งราย (บุคคลธรรมดา / บริษัท / broker)
* Add-on คิดเงินรายใบขนแบบแยกจากแพ็กเกจหลัก (ถ้าจะมีในอนาคต ให้เป็น Phase ถัดไป)
* การคืนเงินอัตโนมัติ / proration ซับซ้อนนอกเหนือจากสถานะ success/failed พื้นฐานของ Payment Gateway

* ชนิดของ ตราสินค้า ปริมาณสินค้า
* เลขที่ใบอนุญาตหรือเอกสารที่เกี่ยวข้อง และเลขที่เอกสารสิทธิประโยชน์ (เช่น Form E / D / RCEP)
